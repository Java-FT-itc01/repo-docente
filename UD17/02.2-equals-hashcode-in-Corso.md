## `equals()` e `hashCode()` nella classe `Corso`

@Override
public boolean equals(Object obj) {
    if (this == obj) {
        return true;
    }

    if (!(obj instanceof Corso)) {
        return false;
    }

    Corso altro = (Corso) obj;
    return codice.equals(altro.codice);
}

@Override
public int hashCode() {
    return codice.hashCode();
}


Questo codice ridefinisce due metodi ereditati da `Object`:

- `equals()`: stabilisce quando due oggetti devono essere considerati uguali.
- `hashCode()`: restituisce un numero intero usato da alcune strutture dati Java per organizzare gli oggetti in memoria.

```java
@Override
public boolean equals(Object obj) {
    if (this == obj) {
        return true;
    }

    if (!(obj instanceof Corso)) {
        return false;
    }

    Corso altro = (Corso) obj;
    return codice.equals(altro.codice);
}

@Override
public int hashCode() {
    return codice.hashCode();
}
````

---

## A cosa serve `equals()`

Il metodo `equals()` viene usato per decidere se due oggetti sono logicamente uguali.

In Java, se non ridefiniamo `equals()`, due oggetti sono uguali solo se sono esattamente lo stesso oggetto in memoria.

Esempio:

```java
Corso c1 = new Corso("JAVA01", "Java Base");
Corso c2 = new Corso("JAVA01", "Java Base");
```

Anche se i due corsi hanno gli stessi dati, per Java sono due oggetti diversi.

Senza `equals()` ridefinito:

```java
c1.equals(c2); // false
```

Con il metodo `equals()` ridefinito come nel codice sopra:

```java
c1.equals(c2); // true
```

Questo perché il confronto non avviene più sull'indirizzo dell'oggetto in memoria, ma sul valore dell'attributo `codice`.

---

## Analisi passo per passo di `equals()`

### 1. Controllo se i due riferimenti puntano allo stesso oggetto

```java
if (this == obj) {
    return true;
}
```

Qui si controlla se l'oggetto corrente e l'oggetto ricevuto come parametro sono lo stesso oggetto in memoria.

Esempio:

```java
Corso c1 = new Corso("JAVA01", "Java Base");

c1.equals(c1); // true
```

In questo caso non serve fare altri controlli.

---

### 2. Controllo se l'oggetto ricevuto è davvero un `Corso`

```java
if (!(obj instanceof Corso)) {
    return false;
}
```

Il metodo `equals()` riceve un parametro di tipo `Object`, quindi potrebbe arrivare qualsiasi cosa:

```java
c1.equals("JAVA01");
c1.equals(null);
c1.equals(new Studente());
```

Questo controllo evita confronti senza senso.

Se `obj` non è un oggetto di tipo `Corso`, il metodo restituisce `false`.

---

### 3. Cast da `Object` a `Corso`

```java
Corso altro = (Corso) obj;
```

Dopo aver verificato che `obj` è davvero un `Corso`, possiamo convertirlo da `Object` a `Corso`.

Questo serve per poter accedere agli attributi specifici della classe `Corso`.

---

### 4. Confronto del codice del corso

```java
return codice.equals(altro.codice);
```

Qui si stabilisce la regola di uguaglianza:

> Due corsi sono considerati uguali se hanno lo stesso codice.

Esempio:

```java
Corso c1 = new Corso("JAVA01", "Java Base");
Corso c2 = new Corso("JAVA01", "Java Avanzato");

c1.equals(c2); // true
```

Anche se il titolo è diverso, i due oggetti vengono considerati uguali perché hanno lo stesso `codice`.

Questa è una scelta progettuale: il codice del corso viene trattato come identificativo univoco.

---

## A cosa serve `hashCode()`

Il metodo `hashCode()` restituisce un numero intero associato all'oggetto.

```java
@Override
public int hashCode() {
    return codice.hashCode();
}
```

In questo caso, il codice hash del corso viene calcolato a partire dall'attributo `codice`.

Esempio:

```java
Corso c1 = new Corso("JAVA01", "Java Base");
Corso c2 = new Corso("JAVA01", "Java Avanzato");
```

Se i due corsi hanno lo stesso `codice`, allora avranno anche lo stesso `hashCode()`.

---

## Perché scrivere `hashCode()` se nel codice non viene chiamato direttamente?

Anche se nel nostro codice non scriviamo mai direttamente:

```java
c1.hashCode();
```

il metodo può essere usato automaticamente da alcune strutture dati Java.

In particolare, viene usato da collezioni come:

```java
HashSet
HashMap
Hashtable
LinkedHashSet
LinkedHashMap
```

Queste strutture usano `hashCode()` per organizzare e ritrovare velocemente gli oggetti.

Quindi `hashCode()` può essere usato anche se noi non lo chiamiamo esplicitamente. Java, con la sua solita discrezione da macchina burocratica, lo usa dietro le quinte.

---

## Esempio con `HashSet`

Supponiamo di avere questo codice:

```java
Set<Corso> corsi = new HashSet<>();

Corso c1 = new Corso("JAVA01", "Java Base");
Corso c2 = new Corso("JAVA01", "Java Avanzato");

corsi.add(c1);
corsi.add(c2);

System.out.println(corsi.size());
```

Dal punto di vista logico, `c1` e `c2` rappresentano lo stesso corso, perché hanno lo stesso codice.

Se `equals()` e `hashCode()` sono entrambi correttamente ridefiniti, il risultato sarà:

```java
1
```

Il secondo corso non viene aggiunto perché il `HashSet` capisce che è già presente un corso con lo stesso codice.

---

## Cosa succede se ridefinisco `equals()` ma non `hashCode()`?

Questo è un errore molto comune.

Se due oggetti sono uguali secondo `equals()`, allora devono avere anche lo stesso `hashCode()`.

Questa è una regola fondamentale di Java.

La regola è:

> Se `a.equals(b)` restituisce `true`, allora `a.hashCode()` e `b.hashCode()` devono restituire lo stesso valore.

Se questa regola non viene rispettata, strutture come `HashSet` e `HashMap` possono comportarsi in modo errato.

Esempio problematico:

```java
Corso c1 = new Corso("JAVA01", "Java Base");
Corso c2 = new Corso("JAVA01", "Java Avanzato");

System.out.println(c1.equals(c2)); // true
```

Se però `hashCode()` non è coerente con `equals()`, un `HashSet` potrebbe trattarli come oggetti diversi.

Risultato possibile:

```java
Set<Corso> corsi = new HashSet<>();

corsi.add(c1);
corsi.add(c2);

System.out.println(corsi.size()); // potrebbe stampare 2
```

E qui il programma non esplode, semplicemente tradisce le aspettative. Che è peggio, perché almeno un'esplosione si nota.

---

## Perché `hashCode()` usa `codice`

Nel metodo `equals()` abbiamo stabilito che due corsi sono uguali se hanno lo stesso `codice`.

Quindi anche `hashCode()` deve basarsi sullo stesso attributo:

```java
@Override
public int hashCode() {
    return codice.hashCode();
}
```

La logica è coerente:

```java
equals()   -> confronta codice
hashCode() -> calcola il codice hash usando codice
```

Questo garantisce che due corsi con lo stesso codice siano trattati correttamente anche nelle collezioni basate su hash.

---

## Versione più robusta con `Objects`

La versione mostrata funziona se `codice` non è mai `null`.

Se però `codice` potesse essere `null`, questo codice potrebbe generare un errore:

```java
return codice.equals(altro.codice);
```

Se `codice` è `null`, si avrebbe una `NullPointerException`.

Una versione più sicura è questa:

```java
import java.util.Objects;

@Override
public boolean equals(Object obj) {
    if (this == obj) {
        return true;
    }

    if (!(obj instanceof Corso)) {
        return false;
    }

    Corso altro = (Corso) obj;
    return Objects.equals(codice, altro.codice);
}

@Override
public int hashCode() {
    return Objects.hash(codice);
}
```

`Objects.equals()` gestisce correttamente anche il caso in cui uno o entrambi i valori siano `null`.

---

## Sintesi finale

In questa classe, `equals()` dice:

> Due oggetti `Corso` sono uguali se hanno lo stesso `codice`.

`hashCode()` serve a mantenere coerente questa regola anche quando gli oggetti vengono usati in strutture come `HashSet` e `HashMap`.

Anche se nel codice non chiamiamo direttamente `hashCode()`, è corretto ridefinirlo perché Java potrebbe usarlo automaticamente nelle collezioni basate su hash.

La coppia `equals()` / `hashCode()` deve sempre essere progettata insieme.

Se si ridefinisce `equals()` senza ridefinire `hashCode()`, il codice può sembrare corretto nei confronti semplici, ma comportarsi male nelle collezioni. In altre parole: funziona finché non serve davvero.
